## 模块一：ArkTS 语言（预计 90 分钟）

### 1.1 语言定位与设计约束

ArkTS 基于 TypeScript，但做了严格的能力收敛。核心设计目标：**支持 AOT 编译以提升运行时性能**。

**保留的能力：**
- 静态类型系统（类型注解、接口、泛型）
- ES6+ 语法：箭头函数、模板字符串、展开运算符、async/await
- 模块化（import/export）
- 装饰器体系

**被移除的能力及原因：**

| 被移除的特性 | 移除原因 |
|-------------|---------|
| `any` / `unknown` | 破坏 AOT 编译的类型确定性，编译器无法生成优化机器码 |
| 运行时的 `delete` 操作 | 破坏对象形状的稳定性，影响内存布局优化 |
| `eval()` / `new Function()` | 需要运行时动态编译，不存在于 ArkCompiler |
| 解构赋值 | 编译器对解构生成中间变量，与方舟编译器的中间表示冲突 |
| `call` / `apply` / `bind` | 动态改变 this 指向与 AOT 的类型推导矛盾 |
| 原型链动态修改 | 破坏类层级结构的稳定性 |

**AOT 编译的执行模型：**

传统的 JS 引擎（V8、JavaScriptCore）采用 JIT（即时编译）：先解释执行，热点代码再编译优化。ArkCompiler 在应用安装阶段就完成编译，运行时直接执行机器码：

```
传统的 JIT 模式：
  源码 → 字节码 → 解释执行 → 热点检测 → 机器码（边跑边编译）

ArkCompiler AOT 模式：
  源码 → 类型推导 → 机器码（安装时一次编译完成）
```

优点是启动速度极快、无预热开销；代价是语言灵活度降低，必须写确定类型的代码。

### 1.2 struct 和 class 的语义区别

这是 ArkTS 设计上的关键区分：

| | struct | class |
|---|--------|-------|
| 用途 | UI 组件定义 | 业务逻辑、数据模型 |
| 是否可继承 | 不可继承 | 可继承 |
| 必须配合的装饰器 | `@Component` | 无强制要求 |
| 核心方法 | `build()` | 无 |
| this 指向 | 组件实例 | 类实例 |
| 内存管理 | 框架管理（随组件树创建/销毁） | 开发者管理（引用计数） |

`struct` 禁止继承是有意为之。传统面向对象 UI 框架（如 Android View 体系）存在深层继承链带来的问题：耦合度高、难以组合、渲染链路不可控。ArkUI 采用组合模式，不支持组件继承，强制开发者通过组合和状态管理来复用行为。

### 1.3 UI 复用三件套：@Styles、@Extend、@Builder

这三个装饰器解决的是同一类问题：消除 UI 代码中的重复模式。

#### @Styles

作用于通用属性集合，定义后通过链式调用附加到任何组件上。不支持参数传递，适合抽取固定的样式序列。

```typescript
@Styles function cardBase() {
  .width('100%')
  .height(120)
  .borderRadius(12)
  .padding(16)
  .backgroundColor(Color.White)
  .shadow({ radius: 8, color: '#1F000000', offsetX: 0, offsetY: 2 })
}

// 使用
Row() { ... }.cardBase()
```

**限制：** @Styles 只能包含通用属性方法。组件私有属性（如 Text 的 `.fontColor()`、Button 的 `.type()`）不能放入 @Styles。

#### @Extend

解决 @Styles 无法处理组件私有属性的问题。@Extend 绑定到特定组件类型，可以访问该组件的全部属性，且支持参数传递。

```typescript
@Extend(Text) function label(size: number, color: ResourceColor) {
  .fontSize(size)
  .fontColor(color)
  .fontWeight(FontWeight.Medium)
  .maxLines(1)
  .textOverflow({ overflow: TextOverflow.Ellipsis })
}

Text('商品标题').label(16, '#333333')
Text('价格').label(20, Color.Red)
```

**与 @Styles 的本质区别：**
- @Styles 是属性级别的复用（"哪些通用属性"）
- @Extend 是组件类型的扩展（"这种组件长什么样"）

#### @Builder

@Builder 解决的是更综合的场景：复用一段完整的 UI 结构，包含组件层级、事件绑定和条件逻辑。它比 @Styles/@Extend 粒度更粗，适用于抽取完整的功能区域。

```typescript
@Builder function EmptyState(icon: Resource, text: string) {
  Column() {
    Image(icon).width(120).height(120).opacity(0.4)
    Text(text).fontSize(14).fontColor('#999999').margin({ top: 16 })
  }
  .justifyContent(FlexAlign.Center)
  .width('100%')
  .height('100%')
}

// 在列表数据为空时使用
if (this.dataList.length === 0) {
  EmptyState($r('app.media.icon_empty'), '暂无数据')
}
```

**@BuilderParam：组件插槽**

@BuilderParam 允许父组件向子组件传递一段 UI 内容作为参数，实现类似"插槽"的效果。

```typescript
@Component
struct Dialog {
  @BuilderParam header: () => void
  @BuilderParam content: () => void
  @BuilderParam footer: () => void

  build() {
    Column() {
      this.header()
      Divider()
      this.content()
      Divider()
      this.footer()
    }
    .backgroundColor(Color.White)
    .borderRadius(16)
  }
}

// 使用时传入三块自定义内容
Dialog() {
  Text('确认操作').fontSize(18).fontWeight(FontWeight.Bold)  // header
} {
  Text('确定要删除这条记录吗？删除后无法恢复。')              // content
} {
  Row() {
    Button('取消')
    Button('确定').backgroundColor(Color.Red)                 // footer
  }
}
```

### 1.4 列表渲染：ForEach vs LazyForEach

两者的差异在于渲染策略和内存占用。这是性能相关的核心知识。

**ForEach：**

全量渲染策略。传入的数组有多少条数据，就创建多少个组件实例。渲染过程发生在一个 build 周期内。

```typescript
ForEach(this.items, (item: Item, index: number) => {
  ListItem() {
    ItemRow({ data: item })
  }
})
```

适用条件：数据量小于 100 条（经验值），且列表项组件不复杂。

**LazyForEach：**

懒加载策略。只创建可视区域及缓冲区的组件实例。滚动时，回收滚出屏幕的组件并复用。

```typescript
class BasicDataSource implements IDataSource {
  private listeners: DataChangeListener[] = []
  private dataArray: string[] = []

  public totalCount(): number {
    return this.dataArray.length
  }

  public getData(index: number): string {
    return this.dataArray[index]
  }

  public addData(index: number, data: string): void {
    this.dataArray.splice(index, 0, data)
    this.notifyDataAdd(index)
  }

  public pushData(data: string): void {
    this.dataArray.push(data)
    this.notifyDataAdd(this.dataArray.length - 1)
  }

  registerDataChangeListener(listener: DataChangeListener): void {
    this.listeners.push(listener)
  }
  unregisterDataChangeListener(listener: DataChangeListener): void {
    const index = this.listeners.indexOf(listener)
    if (index >= 0) this.listeners.splice(index, 1)
  }
  notifyDataReload(): void { this.listeners.forEach(l => l.onDataReloaded()) }
  notifyDataAdd(index: number): void { this.listeners.forEach(l => l.onDataAdd(index)) }
  notifyDataChange(index: number): void { this.listeners.forEach(l => l.onDataChange(index)) }
  notifyDataDelete(index: number): void { this.listeners.forEach(l => l.onDataDelete(index)) }
}

// 使用
List() {
  LazyForEach(this.dataSource, (item: string) => {
    ListItem() {
      Text(item)
    }
  }, (item: string) => JSON.stringify(item))  // keyGenerator，必须提供
}
```

**关键细节：**
- `keyGenerator` 必须返回唯一且稳定的标识。如果 key 相同，框架会复用已创建的组件实例。
- IDataSource 的 `notifyDataChange` 方法触发的是局部更新，不是全量重建。
- LazyForEach 必须用在容器组件（List、Grid、Swiper、WaterFlow）内部。

## 模块二：状态管理（预计 90 分钟）

### 2.1 状态驱动的核心流程

ArkUI 采用声明式 UI 范式，开发者描述"状态和 UI 的映射关系"，框架负责"当状态变化时执行最小化 UI 更新"。完整数据流：

```
用户交互 / 异步回调 / 定时器
        │
        ▼
  修改 @State 变量
        │
        ▼
  装饰器运行时检测到变化
        │
        ▼
  标记组件为 "dirty"
        │
        ▼
  调度重渲染（合并同一帧内的多次状态变更）
        │
        ▼
  执行 build() → 生成新的 UI 描述
        │
        ▼
  Diff 新旧 UI 树 → 仅更新变化的节点
```

### 2.2 核心装饰器深度对比

#### @State

组件自身的状态。被 @State 修饰的变量变化时，组件重新执行 build()。

```typescript
@Component
struct Counter {
  @State count: number = 0

  build() {
    Column() {
      Text(`计数：${this.count}`)
      Button('增加')
        .onClick(() => {
          this.count++  // 触发重渲染
        })
    }
  }
}
```

**检测机制：** @State 使用属性代理（Property Proxy）。框架拦截赋值操作，比较新值与旧值（浅比较）。对于基本类型，值不同则标记 dirty；对于引用类型，引用变化才标记 dirty。

**关键陷阱：嵌套对象属性的修改不会被检测到。**

```typescript
@State user: User = { name: 'Zhang', age: 25 }

// ❌ 这不会触发 UI 更新
this.user.age = 26

// ✅ 需要替换整个对象引用
this.user = { ...this.user, age: 26 }

// ✅ 或使用 @Observed + @ObjectLink 处理深层变化
```

#### @Prop

父组件向子组件的单向数据传递。子组件接收的是父组件值的拷贝，修改不影响父组件。

```typescript
@Component
struct ChildView {
  @Prop message: string = ''

  build() {
    Text(this.message)
  }
}

// 父组件
ChildView({ message: this.parentMessage })
```

**数据流向：** 父 → 子，单向，不可逆。

**更新机制：** 当父组件中 `parentMessage` 变化时，触发父组件重渲染。父组件的 build() 重新执行时，将新值传给子组件的 `@Prop`，子组件随之重渲染。

**适用场景：** 展示型子组件，子组件不需要修改的值。

#### @Link

父子组件的双向数据同步。子组件修改会反向影响父组件。

```typescript
@Component
struct ToggleChild {
  @Link checked: boolean

  build() {
    Toggle({ type: ToggleType.Switch, isOn: this.checked })
      .onChange((value: boolean) => {
        this.checked = value  // 修改会同步回父组件
      })
  }
}

// 父组件：传值时加 $ 前缀
ToggleChild({ checked: $parentChecked })
```

**关键约束：**
- `@Link` 变量不能初始化，值由父组件提供
- 父组件传值时必须使用 `$` 操作符（语法层面表示传递引用）
- 双向同步基于引用语义实现，不是数据拷贝

**适用场景：** 开关、滑块、输入框等需要反向修改的交互组件。

#### @Provide 和 @Consume

跨层级状态共享，解决"爷爷传孙子"要经过中间层的问题。

```typescript
// 提供方（可以是任意层级）
@Component
struct AncestorView {
  @Provide('themeColor') theme: string = 'light'

  build() {
    Column() {
      IntermediateView()  // 中间层不传参
    }
  }
}

// 消费方（任意后代）
@Component
struct DeepChildView {
  @Consume('themeColor') theme: string

  build() {
    Text(`当前主题：${this.theme}`)
  }
}
```

**工作机制：**
- `@Provide` 将数据注册到组件树的提供者链上
- `@Consume` 沿组件树向上查找最近的同名提供者
- 提供者的数据变化时，所有消费者同步更新

**同名覆盖规则：** 内层 @Provide 会遮蔽外层同名的 @Provide，最近的生效。

#### @Observed 和 @ObjectLink

解决 @State 无法检测嵌套对象属性变化的限制。

```typescript
@Observed
class UserProfile {
  name: string
  age: number
  address: Address

  constructor(name: string, age: number, address: Address) {
    this.name = name
    this.age = age
    this.address = address
  }
}

@Observed
class Address {
  city: string
  street: string

  constructor(city: string, street: string) {
    this.city = city
    this.street = street
  }
}

@Component
struct ProfileCard {
  @ObjectLink user: UserProfile

  build() {
    Column() {
      Text(`姓名：${this.user.name}`)
      Text(`年龄：${this.user.age}`)
      Text(`地址：${this.user.address.city} ${this.user.address.street}`)
      Button('修改城市')
        .onClick(() => {
          this.user.address.city = 'Shanghai'  // ✅ 多层嵌套也能检测到
        })
    }
  }
}
```

**工作机制：**
- `@Observed` 在类构造时为每个属性设置 getter/setter 代理
- 属性修改时，触发关联的 `@ObjectLink` 持有的组件更新
- 支持多层嵌套：Address 也需要 @Observed，否则 address.city 的变化无法追踪

**更新粒度：** 精确到被修改的那个 @ObjectLink 组件，不是刷新整个父组件。

**版本演进：** API 12 引入了状态管理 V2（`@ObservedV2` / `@Trace`），进一步优化：
- `@Trace` 可标记类的特定属性，实现更细粒度的变化追踪
- 不再需要 `@ObjectLink`，直接在组件中使用 `@ObservedV2` 类即可
- 减少不必要的组件重渲染

### 2.3 全局状态：AppStorage / LocalStorage / PersistentStorage

#### AppStorage

应用全局的单例状态容器。所有 UIAbility 和页面共享。

```typescript
// 设置
AppStorage.setOrCreate<string>('currentUser', 'user_123')

// 获取
const userId = AppStorage.get<string>('currentUser')

// 组件内绑定（自动订阅变化）
@StorageLink('currentUser') userId: string = ''

// 单向绑定（只在初始化时获取一次）
@StorageProp('currentUser') userId: string = ''
```

**注意：**
- 数据生命周期与进程一致，进程被杀后数据丢失
- 所有页面共享同一份数据，修改会通知所有绑定的组件
- 适合：登录状态、全局配置、主题模式

#### LocalStorage

页面级或 UIAbility 级的状态容器。不同页面/Ability 的 LocalStorage 相互隔离。

```typescript
// 在 UIAbility 创建时注入
let storage = new LocalStorage()
storage.setOrCreate<number>('currentTab', 0)

windowStage.loadContent('pages/Index', storage)

// 页面内绑定
@LocalStorageLink('currentTab') tabIndex: number = 0
@LocalStorageProp('currentTab') tabIndex: number = 0
```

**与 AppStorage 的区别：**
- AppStorage 全局唯一，LocalStorage 可按需创建多个实例
- LocalStorage 适合多窗口场景：每个窗口拥有独立的 LocalStorage
- 关闭窗口时对应的 LocalStorage 销毁

#### PersistentStorage

将 AppStorage 中的指定 key 持久化到磁盘，应用重启后自动恢复。

```typescript
// 将 AppStorage 中的 'fontSize' 与磁盘持久化关联
PersistentStorage.persistProp('fontSize', 16)

// 此后对 AppStorage 中 'fontSize' 的修改会自动落盘
AppStorage.set<number>('fontSize', 18)  // 自动写入磁盘
```

**关键限制：**
- PersistentStorage 不是通用的持久化方案，它只是 AppStorage 的磁盘同步层
- 数据存储在系统管理的 XML 文件中，大小限制约 8KB
- 写入操作发生在主线程，频繁写入或写入大量数据会导致卡顿
- **不要用于存储列表、对象数组等结构化数据，这些应该使用 RDB 或 Preferences**

## 模块三：Stage 模型与生命周期（预计 60 分钟）

### 3.1 UIAbility 完整生命周期

UIAbility 是包含 UI 界面的应用组件，是系统调度的基本单元。

```
                    应用启动
                       │
                       ▼
              ┌─────────────────┐
              │    onCreate()    │  初始化全局服务、注册回调
              └────────┬────────┘
                       │
                       ▼
              ┌─────────────────────────┐
              │ onWindowStageCreate()   │  窗口创建，加载页面
              │  → loadContent(page)    │  (必须在此回调中完成)
              └────────┬────────────────┘
                       │
                       ▼
              ┌─────────────────┐
              │  onForeground()  │  进入前台，恢复交互
              └────────┬────────┘
                       │
              ┌────────┴────────┐
              │   前台运行中      │
              └────────┬────────┘
                       │
                       ▼
              ┌─────────────────┐
              │  onBackground()  │  退到后台，暂停非必要任务
              └────────┬────────┘
                       │     ← 可能重新回到 onForeground
                       ▼
              ┌──────────────────────────┐
              │ onWindowStageDestroy()   │  窗口销毁
              └────────┬─────────────────┘
                       │
                       ▼
              ┌─────────────────┐
              │   onDestroy()    │  释放所有资源
              └─────────────────┘
```

**各阶段应执行的操作：**

| 回调 | 应执行 | 不应执行 |
|------|--------|---------|
| onCreate | 初始化全局服务、创建数据库连接、读取配置 | 加载 UI（窗口尚未创建） |
| onWindowStageCreate | 调用 loadContent 加载页面、设置窗口属性 | 耗时异步操作（会延迟首屏显示） |
| onForeground | 恢复动画、重新注册传感器、刷新数据 | 大量数据预加载 |
| onBackground | 暂停动画、释放相机/定位等敏感资源 | 执行耗时任务（后台很快被冻结） |
| onDestroy | 关闭数据库、取消订阅、释放内存 | 启动新的异步操作（Ability 即将销毁） |

### 3.2 UIAbility 启动模式

三种启动模式决定同一个 Ability 是否可以存在多个实例：

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| singleton（默认） | 系统中只能存在一个实例 | 主界面、设置页 |
| standard | 每次启动创建新实例 | 文档编辑页（多文档同时编辑） |
| specified | 通过 key 决定是创建新实例还是复用已有实例 | 浏览器标签页（同一域名复用） |

在 `module.json5` 中配置：
```json
{
  "abilities": [{
    "name": "EntryAbility",
    "launchType": "singleton"
  }]
}
```

### 3.3 Context 的层级

Context 是访问系统资源和能力的入口，存在层级关系：

```
ApplicationContext（全局唯一）
    ├── UIAbilityContext（每个 UIAbility 一个）
    │     ├── 获取窗口信息
    │     ├── 启动其他 Ability
    │     └── 权限申请
    ├── ServiceExtensionContext
    └── FormExtensionContext
```

**ApplicationContext vs UIAbilityContext：**

- `ApplicationContext`：全局生命周期，应用存在期间一直可用。适合注册全局事件监听、管理全局状态
- `UIAbilityContext`：绑定到具体 Ability，Ability 销毁后失效。适合页面跳转、弹窗、获取当前窗口

```typescript
// 获取 ApplicationContext
let appContext = this.context.getApplicationContext()

// UIAbilityContext（在 UIAbility 内部）
let abilityContext = this.context

// 典型用法区别
appContext.on('screenLock', callback)  // 全局事件
abilityContext.startAbility(want)       // 页面跳转
```

### 3.4 ExtensionAbility 简介

ExtensionAbility 是面向特定场景的功能组件，不包含 UI，由系统管理生命周期。

| 类型 | 用途 |
|------|------|
| FormExtensionAbility | 桌面卡片/服务卡片 |
| WorkSchedulerExtensionAbility | 后台任务调度 |
| InputMethodExtensionAbility | 输入法 |

| EnterpriseAdminExtensionAbility | 企业设备管理 |

### 3.5 路由系统：Router vs Navigation

#### Router（旧方案）

基于页面栈的路由管理，页面以 Page 为单位进行跳转。

```typescript
import { router } from '@kit.ArkUI'

// 跳转
router.pushUrl({
  url: 'pages/Detail',
  params: { id: 123 }
})

// 返回
router.back()

// 替换当前页面
router.replaceUrl({ url: 'pages/Login' })

// 获取参数
const params = router.getParams()
```

**Router 的局限性：**
- 跳转粒度为页面级，不支持页面内部组件级路由
- 不支持分栏布局，无法在折叠屏展开时显示"列表+详情"双栏
- 不支持交互式转场动画（如 iOS 的滑动返回）
- 页面栈是全局的，管理复杂容易出 bug

#### Navigation（推荐方案）

组件级路由，通过 Navigation 容器组件和 NavPathStack 管理页面栈。**这是 HarmonyOS NEXT 官方推荐的路由方案。**

```typescript
@Entry
@Component
struct MainPage {
  @Provide navPathStack: NavPathStack = new NavPathStack()

  @Builder
  pageMap(name: string, param: Object) {
    if (name === 'Detail') {
      DetailPage({ id: param['id'] as number })
    } else if (name === 'Search') {
      SearchPage()
    }
  }

  build() {
    Navigation(this.navPathStack) {
      Column() {
        Button('查看详情')
          .onClick(() => {
            this.navPathStack.pushPathByName('Detail', { id: 123 })
          })
      }
    }
    .navDestination(this.pageMap)
    .mode(NavigationMode.Auto)  // Auto = 根据设备自动切换分栏/不分栏
  }
}

// 目标页面：必须标记 @Entry 或使用 NavDestination
@Component
struct DetailPage {
  @Consume navPathStack: NavPathStack
  @State id: number = 0

  build() {
    NavDestination() {
      Column() {
        Text(`详情页 ID: ${this.id}`)
        Button('返回')
          .onClick(() => {
            this.navPathStack.pop()
          })
      }
    }
    .title('详情')
  }
}
```

**Navigation 的优势：**
- 组件级路由：页面内部也可以包含子路由
- 分栏模式：折叠屏展开时自动切换为左右分栏（`NavigationMode.Split`）
- 自定义转场动画：支持通过 `customNavContentTransition` 定义动画
- 页面栈隔离：每个 Navigation 拥有独立的路径栈

### 4.1 HTTP 网络请求

使用 `@kit.NetworkKit` 中的 http 模块，基于 Promise 的异步 API。

```typescript
import { http } from '@kit.NetworkKit'

async function requestData(url: string): Promise<object> {
  const httpRequest = http.createHttp()

  try {
    const response = await httpRequest.request(url, {
      method: http.RequestMethod.GET,
      header: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${AppStorage.get<string>('token')}`
      },
      connectTimeout: 10000,
      readTimeout: 30000
    })

    if (response.responseCode === 200) {
      return JSON.parse(response.result as string)
    }

    // 处理不同状态码
    if (response.responseCode === 401) {
      // token 过期，跳转登录
    }
    throw new Error(`HTTP Error: ${response.responseCode}`)
  } catch (err) {
    console.error('请求失败:', JSON.stringify(err))
    throw err
  } finally {
    httpRequest.destroy()  // 防止内存泄漏，必须在 finally 中调用
  }
}
```

**封装网络层的关键点：**

1. 自动注入 token，token 过期后自动刷新
2. 统一错误处理（网络异常、服务端错误码、数据解析失败）
3. 请求/响应拦截
4. 并发控制

```typescript
class HttpClient {
  private baseUrl: string
  private pendingRefresh: Promise<string> | null = null

  constructor(baseUrl: string) {
    this.baseUrl = baseUrl
  }

  async request<T>(method: http.RequestMethod, path: string, body?: object): Promise<T> {
    const httpRequest = http.createHttp()
    try {
      let token = AppStorage.get<string>('token') || ''

      const response = await httpRequest.request(`${this.baseUrl}${path}`, {
        method,
        header: {
          'Authorization': `Bearer ${token}`,
          'Content-Type': 'application/json'
        },
        extraData: body ? JSON.stringify(body) : undefined,
        connectTimeout: 10000,
        readTimeout: 30000
      })

      if (response.responseCode === 200) {
        return JSON.parse(response.result as string) as T
      }

      throw new ApiError(response.responseCode, response.result as string)
    } finally {
      httpRequest.destroy()
    }
  }

  get<T>(path: string): Promise<T> {
    return this.request<T>(http.RequestMethod.GET, path)
  }

  post<T>(path: string, body: object): Promise<T> {
    return this.request<T>(http.RequestMethod.POST, path, body)
  }
}
```

**并发请求处理：**

```typescript
// Promise.all 实现并发
async function loadPageData(userId: string) {
  const [profile, orders, notifications] = await Promise.all([
    api.get<UserProfile>(`/user/${userId}`),
    api.get<Order[]>(`/orders?userId=${userId}`),
    api.get<Notification[]>(`/notifications`)
  ])
  return { profile, orders, notifications }
}

// async/await 串行（有依赖关系时）

```

### 4.2 数据持久化方案

鸿蒙提供四种数据持久化方案，按数据结构复杂度递增：

| 方案 | 内核 | 数据模型 | SQL支持 | 跨设备同步 | 适用场景 |
|------|------|---------|---------|-----------|---------|
| Preferences | 本地KV存储 | KV | 无 | 无 | 用户偏好、简单配置 |
| KV-Store | 分布式KV | KV | 无 | ✅ | 多设备共享的简单数据 |
| RDB | SQLite | 关系型 | ✅ | 无 | 结构化数据、复杂查询 |
| 文件 | 文件系统 | 文件 | 无 | 无 | 图片、日志、大文件 |

#### Preferences

本质是对系统级 Key-Value 存储的封装，数据保存在 XML 文件中。

```typescript
import { preferences } from '@kit.ArkData'

// 获取实例（name 区分不同存储空间）
const prefs = await preferences.getPreferences(this.context, 'app_settings')

// 写入
await prefs.put('token', 'xxxxx')
await prefs.put('isFirstLaunch', false)
await prefs.flush()  // 必须调用 flush 才真正写入磁盘

// 读取（defaultValue 在 key 不存在时返回）
const token = await prefs.get('token', '')
const isFirst = await prefs.get('isFirstLaunch', true) as boolean

// 删除
await prefs.delete('token')
await prefs.flush()

// 监听变化
prefs.on('change', (key: string) => {
  console.log(`key ${key} 发生变化`)
})
```

**注意事项：**
- `flush()` 会触发磁盘写入，不要频繁调用。建议在关键操作后（如登录成功、退出登录）调用一次
- 不支持存储复杂数据结构，如有需求应使用 RDB
- 每个实例对应一个 xml 文件，不要创建过多实例

#### RDB（Relational Database）

基于 SQLite 的关系型数据库，支持 SQL 查询。

```typescript
import { relationalStore } from '@kit.ArkData'

// 1. 数据库配置
const STORE_CONFIG: relationalStore.StoreConfig = {
  name: 'main.db',
  securityLevel: relationalStore.SecurityLevel.S1
}

// 2. 获取数据库实例
const rdbStore = await relationalStore.getRdbStore(this.context, STORE_CONFIG)

// 3. 创建表
const CREATE_USER_TABLE = `
  CREATE TABLE IF NOT EXISTS user (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    phone TEXT,
    email TEXT UNIQUE,
    created_at INTEGER DEFAULT (strftime('%s', 'now'))
  )
`
await rdbStore.executeSql(CREATE_USER_TABLE)

// 4. 插入数据
const user = {
  name: '张三',
  phone: '13800000000',
  email: 'zhang@example.com'
}
const rowId = await rdbStore.insert('user', user as relationalStore.ValuesBucket)

// 5. 查询数据（使用 Predicates 构建查询条件）
let predicates = new relationalStore.RdbPredicates('user')
predicates.like('name', '%张%')
  .and()
  .orderByDesc('created_at')
  .limit(20)

const resultSet = await rdbStore.query(predicates, ['id', 'name', 'email'])

// 遍历结果集
const users: Array<Record<string, string | number>> = []
while (resultSet.goToNextRow()) {
  users.push({
    id: resultSet.getLong(resultSet.getColumnIndex('id')),
    name: resultSet.getString(resultSet.getColumnIndex('name')),
    email: resultSet.getString(resultSet.getColumnIndex('email'))
  })
}
resultSet.close()

// 6. 更新数据
const updateValues: relationalStore.ValuesBucket = { phone: '13900000000' }
let updatePredicates = new relationalStore.RdbPredicates('user')
updatePredicates.equalTo('id', rowId)
await rdbStore.update(updateValues, updatePredicates)

// 7. 删除数据
let deletePredicates = new relationalStore.RdbPredicates('user')
deletePredicates.equalTo('id', rowId)
await rdbStore.delete(deletePredicates)

// 8. 事务
rdbStore.beginTransaction()
try {
  await rdbStore.insert('user', user1 as relationalStore.ValuesBucket)
  await rdbStore.insert('user', user2 as relationalStore.ValuesBucket)
  rdbStore.commit()
} catch (e) {
  rdbStore.rollBack()
}
```

**Predicates 常用条件：**

```typescript
const p = new relationalStore.RdbPredicates('table')

p.equalTo('status', 'active')           // column = value
p.notEqualTo('status', 'deleted')       // column != value
p.greaterThan('age', 18)                // column > value
p.lessThanOrEqualTo('age', 60)          // column <= value
p.between('age', 18, 60)                 // column BETWEEN v1 AND v2
p.like('name', '%keyword%')              // column LIKE pattern
p.inValues('id', [1, 2, 3])             // column IN (...)
p.isNull('deleted_at')                  // column IS NULL
p.and()                                  // 组合使用 AND
p.or()                                   // 组合使用 OR
p.orderByAsc('name')                     // 升序
p.orderByDesc('created_at')             // 降序
p.limit(10)                              // 限制数量
p.offset(20)                             // 偏移
p.groupBy(['category'])                  // 分组
p.distinct()                             // 去重
```

### 5.1 断点系统

断点系统是鸿蒙响应式布局的基础。系统根据窗口宽度自动划分断点，应用通过监听断点变化来切换布局。

| 断点 | 宽度范围 (vp) | 典型设备形态 |
|------|-------------|------------|
| xs | 0 ≤ width < 320 | 极小屏设备 |
| sm | 320 ≤ width < 600 | 手机竖屏（绝大多数手机） |
| md | 600 ≤ width < 840 | 手机横屏、折叠屏展开、小平板竖屏 |
| lg | 840 ≤ width | 大平板、PC/2in1 |

**vp（virtual pixel）说明：** vp 是鸿蒙的虚拟像素单位，类似 Android 的 dp 和 iOS 的 pt。以 160dpi 为基准，1vp ≈ 1 物理像素在 160dpi 屏幕上。这种抽象确保相同 vp 值在不同密度屏幕上物理尺寸一致。

**获取当前断点：**

```typescript
import { window } from '@kit.ArkUI'
import { BreakpointType } from '@ohos.breakpoint'

@Entry
@Component
struct AdaptivePage {
  @State breakpoint: string = 'sm'
  private onBreakpointChange = (breakpoint: string) => {
    this.breakpoint = breakpoint
  }

  aboutToAppear() {
    window.getLastWindow(this.context).then((win) => {
      win.on('windowSizeChange', (data) => {
        const width = data.width
        if (width < 320) this.breakpoint = 'xs'
        else if (width < 600) this.breakpoint = 'sm'
        else if (width < 840) this.breakpoint = 'md'
        else this.breakpoint = 'lg'
      })
    })
  }

  build() {
    if (this.breakpoint === 'sm' || this.breakpoint === 'xs') {
      SingleColumnLayout()
    } else {
      TwoColumnLayout()
    }
  }
}
```

### 5.4 包类型：HAP / HAR / HSP

鸿蒙应用的模块化体系定义了三种包类型，核心区别在于代码共享策略：

```
┌─────────────────────────────────────────────────┐
│                   App 包                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐        │
│  │ Entry    │  │ Feature  │  │ Feature  │        │
│  │ HAP      │  │ HAP      │  │ HAP      │        │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘        │
│       │    ┌───────┴──────┐       │              │
│       └────┤     HSP      ├───────┘              │
│            │  (动态共享包) │                       │
│            └──────────────┘                       │
│  ┌──────────────┐  ┌──────────────┐              │
│  │   HAR        │  │   HAR        │              │
│  │ (静态共享包) │  │ (静态共享包) │              │
│  └──────────────┘  └──────────────┘              │
└─────────────────────────────────────────────────┘
```

| 类型 | 全称 | 独立性 | 共享方式 | 何时使用 |
|------|------|--------|---------|---------|
| HAP | Harmony Ability Package | 可独立安装的模块 | 不共享，每个 HAP 包含完整代码 | Entry 模块、Feature 模块 |
| HAR | Harmony Archive | 不可独立安装，必须被 HAP 引用 | 静态共享：编译时将代码复制到每个引用它的 HAP | 工具库、公共 UI 组件 |
| HSP | Harmony Shared Package | 不可独立安装，被多个 HAP 引用 | 动态共享：运行时多个 HAP 共享同一份代码和资源 | 被多个 HAP 引用的核心库 |

**HAR vs HSP 的选型判断：**

应当根据使用频率和体积综合决策。若一个模块被多个 HAP 引用且模块体积较大（如一个包含大量组件的 UI 库），使用 HSP 可避免重复打包，显著减小应用体积。若模块体积小（如一个纯工具函数库），使用 HAR 足够简便。

### 5.5 权限管理

鸿蒙采用分级权限模型。以 `ohos.permission.LOCATION`（位置权限）为例：

```typescript
import { abilityAccessCtrl, Permissions } from '@kit.AbilityKit'

// 步骤 1：module.json5 中声明权限
// {
//   "module": {
//     "requestPermissions": [
//       { "name": "ohos.permission.LOCATION" },
//       { "name": "ohos.permission.CAMERA" }
//     ]
//   }
// }

async function requestPermission(permission: Permissions): Promise<boolean> {
  const atManager = abilityAccessCtrl.createAtManager()
  const tokenId = bundleManager.getBundleInfoForSelfSync(
    bundleManager.BundleFlag.GET_BUNDLE_INFO_WITH_APPLICATION
  ).appInfo.accessTokenId

  // 步骤 2：检查权限状态
  const grantStatus = atManager.checkAccessTokenSync(tokenId, permission)

  if (grantStatus === abilityAccessCtrl.GrantStatus.PERMISSION_GRANTED) {
    return true
  }

  // 步骤 3：向用户请求权限
  const result = await atManager.requestPermissionsFromUser(
    this.context,
    Array.from([permission])
  )

  return result.authResults[0] === abilityAccessCtrl.GrantStatus.PERMISSION_GRANTED
}
```

**权限级别分类：**
- `system_grant`：安装时自动授予（网络、振动等）
- `user_grant`：需用户手动同意（位置、相机、麦克风等）
- `system_core`：仅系统应用可申请

## 模块六：高频面试题（预计 90 分钟）

以下问题的回答覆盖了面试官期望听到的核心要点。

### Q1：@State、@Prop、@Link 的区别？

**必考程度：很高**

> @State 是组件内部的状态变量，修饰的变量变化时触发组件重新执行 build()。检测机制是浅比较——基本类型比较值，引用类型比较引用地址。因此直接修改对象属性不会触发更新，需要整体替换对象引用。
>
> @Prop 是父传子的单向数据流。子组件接收的是父组件值的拷贝，子组件的修改不会回传父组件。当父组件重渲染时，@Prop 的值会更新为最新值。
>
> @Link 是父子双向同步。底层基于引用语义，父组件传值时必须使用 `$` 前缀标识。子组件修改 @Link 变量后，父组件的源变量同步更新。@Link 变量不能有初始值，必须在父组件通过 `$` 传值。
>
> 总结：@State 管自己，@Prop 单向传入，@Link 双向绑定。

### Q2：ForEach 和 LazyForEach 的区别与选型？

**必考程度：很高**

> ForEach 在 build() 执行时全量生成所有数据的组件实例。如果数组有 1000 条，就生成 1000 个组件节点。优点是实现简单，数组直接传入即可。缺点是数据量大时占用大量内存，首帧渲染耗时长。
>
> LazyForEach 只创建可视区域及预加载缓冲区的组件实例。滚动时组件被回收复用，内存占用稳定在 O(可视区域)。需要实现 IDataSource 接口来提供数据访问和变更通知。三个参数：数据源对象、渲染函数、keyGenerator。keyGenerator 返回每条数据的唯一标识，框架据此判断组件复用关系。
>
> 选型原则：数据量 < 100 且组件不复杂的场景用 ForEach；长列表、无限滚动、图文混排等场景必须用 LazyForEach。

### Q3：UIAbility 的生命周期是什么？各阶段应该做什么？

**必考程度：高**

> UIAbility 有四个核心生命周期状态：Create、Foreground、Background、Destroy。
>
> onCreate：能力创建时触发。应在此阶段完成全局资源初始化，如创建数据库连接、加载配置。注意此时窗口尚未创建，不能操作 UI。
>
> onWindowStageCreate：窗口创建后触发。必须在此回调中调用 loadContent() 加载页面。可以设置窗口属性（全屏、方向锁定等）。
>
> onForeground：从后台回到前台时触发。应恢复动画、重新注册传感器、刷新数据。
>
> onBackground：从前台进入后台时触发。应暂停动画、释放相机/定位等耗电资源。鸿蒙对后台限制严格，应用可能在数秒后被挂起。
>
> onDestroy：Ability 即将销毁时触发，应用场景为释放所有资源，取消事件监听，关闭数据库连接。

### Q4：Router 和 Navigation 的区别？当前官方推荐哪个？

**必考程度：高**

> Router 是早期版本的路由方案，基于全局页面栈。通过 pushUrl/pop 方法管理页面进出。其局限性在于：路由粒度是页面级，不能在页面内做子路由；不支持分栏布局，折叠屏展开时无法同时显示两个页面；转场动画不可定制。
>
> Navigation 是 HarmonyOS NEXT 推荐的组件级路由方案。核心组件是 Navigation 容器和 NavPathStack。通过 pushPathByName/pop 操作栈，每个 Navigation 拥有独立的路径栈，互不干扰。关键优势是支持分栏模式：折叠屏展开时自动显示左右双栏（通过 NavigationMode.Split）。同时支持自定义转场动画和交互式手势返回。
>
> 目前官方文档已明确推荐新项目使用 Navigation，Router 主要用于兼容旧版本。

### Q5：鸿蒙有哪些数据存储方案？各自适用什么场景？

**必考程度：高**

> 四种方案，按数据复杂度递增：
>
> Preferences：Key-Value 键值对存储。适合保存少量轻量配置：登录 token、主题模式、字体大小。数据保存在 XML 文件，通过 flush() 同步到磁盘。注意 flush() 是同步磁盘操作，频繁调用会有性能问题。
>
> KV-Store：在 Preferences 基础上增加了分布式能力。数据可以在同一账号的多台设备间自动同步。适合需要跨设备共享的简单配置。
>
> RDB：基于 SQLite 的关系型数据库。支持 SQL 查询、事务、索引。适合存储有复杂查询需求的结构化数据：用户信息、订单记录、本地缓存。通过 RdbPredicates 构建查询条件，支持联表查询、分组聚合。
>
> File：直接操作文件系统。适合存储图片、音视频、日志等非结构化数据。

### Q6：PersistentStorage 和 RDB 的区别？

**必考程度：中高**

> PersistentStorage 本质不是独立的存储方案。它的作用是自动将 AppStorage 中的指定 key 持久化到磁盘，并在应用启动时自动恢复。底层是系统管理的 XML 文件。
>
> 关键限制：数据通过 AppStorage 中转，修改频繁时会反复序列化到磁盘，且操作发生在主线程。因此只能存储简单值类型（字符串、数字、布尔），不适合存储数组或对象。
>
> RDB 是独立的数据库引擎，支持结构化数据存储和复杂查询。增删改查有独立的事务和索引支持，不在主线程执行存储，性能远优于 PersistentStorage 用于复杂数据。
>
> 简单结论：App 重启后需要记住的几个简单配置用 PersistentStorage，有结构化数据和查询需求用 RDB。

### Q7：TaskPool 和 Worker 的区别？

**必考程度：中**

> TaskPool 和 Worker 都基于鸿蒙的 Actor 线程模型。核心特点是线程间内存不共享，通过消息传递（序列化）通信，从根上消除了锁竞争和死锁。
>
> TaskPool：由系统管理的线程池，开发者不需要手动创建/销毁线程。将任务函数提交给 TaskPool，系统自动分配线程执行。适合短时间的独立计算任务（JSON 解析、图片解码）。优点是不需要管理线程生命周期，系统自动调优并发度。
>
> Worker：开发者手动创建并管理生命周期的独立线程。一个进程中 Worker 数量有上限（约 8 个）。适合需要长期驻留的后台任务（实时数据处理、长连接维护）。需要手动调用 terminate() 释放。
>
> 选型：无特殊要求优先用 TaskPool；需要长时间存活、维持状态的任务用 Worker。

### Q8：HAP、HAR、HSP 的区别？

**必考程度：中**

> 三者在模块化程度和代码共享策略上有本质区别。
>
> HAP 是应用的功能模块，可以独立安装运行。一个应用至少有一个 Entry 类型的 HAP。Feature 类型的 HAP 可以按需动态加载。
>
> HAR 是静态共享包。编译时 HAR 的代码和资源会被完整复制到每个引用它的 HAP 中。如果两个 HAP 都引用了同一个 HAR，最终包体中会存在两份 HAR 代码。优点是简单直接，HAP 可以独立运行；缺点是包体积冗余。
>
> HSP 是动态共享包。运行时多个 HAP 共享同一份 HSP 的代码和资源，不会被复制。HSP 随应用主包一起分发，不能独立安装。多个 HAP 共用且体积较大的代码库使用 HSP 能显著减小包体。
>
> 选型原则：独立功能模块 → HAP；小工具函数 → HAR；多模块共用的大库 → HSP。

### Q9：嵌套对象属性修改不触发 UI 更新怎么办？

**必考程度：高**

> 这是 @State 装饰器的已知限制：基于浅比较检测变化，只追踪引用地址是否改变。如果修改了 @State 修饰的对象的内部属性，引用地址不变，框架认为没有变化。
>
> 解决方案一（简单场景）：修改属性后整体替换对象引用：`this.obj = { ...this.obj, prop: newValue }`。
>
> 解决方案二（复杂嵌套）：使用 @Observed + @ObjectLink。给数据类添加 @Observed 装饰器，框架为该类的每个属性设置 getter/setter 代理。子组件使用 @ObjectLink 接收该对象，即可监听到任意深度的属性变化。多层嵌套需要每一层的类都标记 @Observed。
>
> 解决方案三（API 12+）：使用状态管理 V2 的 @ObservedV2 + @Trace。@Trace 可以标记类中的特定属性，实现比 @Observed 更精细的追踪粒度。

### Q10：沉浸式状态栏如何实现？

**必考程度：中低**

> 调用 `window.setWindowLayoutFullScreen(true)` 使应用内容延伸到状态栏和导航栏区域。但此时内容可能被系统栏遮挡，需要通过系统安全区域接口（`window.getLastWindow().getWindowAvoidArea()`）获取各方向的安全距离，然后给被遮挡的内容区域设置对应的 Padding。

### Q11：if/else 和 Visibility.None 的区别？

**必考程度：中**

> 这个问题本质上是"销毁重建"和"隐藏"的区别。
>
> 使用 if/else 切换时，条件为假的分支组件会被完全销毁，从组件树上移除。条件重新为真时，组件会被重新创建，经历完整的生命周期（aboutToAppear → build）。代价较大，但在不需要时释放了内存。
>
> 使用 Visibility.None 时，组件节点保留在组件树中，只是不参与绘制和测量。切换回 Visible 时很快，因为不需要重建，但组件及其状态一直占用内存。
>
> 频繁切换显隐用 Visibility.None 开销更小；条件根本性变化或需要释放内存时用 if/else。

### Q12：鸿蒙如何实现一次开发多端适配？

**必考程度：中高**

> 鸿蒙多设备适配通过三个层面的能力协同实现：
>
> 1. 自适应布局：使用百分比、vp 单位、Flex 弹性布局，让组件根据容器尺寸自动调整。Column/Row 等容器组件天然支持自适应排版。
>
> 2. 响应式布局：基于断点系统（xs/sm/md/lg）识别设备形态，结合 GridRow/GridCol 栅格布局实现不同断点下的差异化列数和排版。断点变化时，框架自动重新布局，开发者只需在声明中定义各断点下的 span 值。
>
> 3. 资源定制：通过资源限定词目录（phone/tablet/dark 等），系统根据当前设备特征自动选择最匹配的资源文件，无需代码分支判断。
>
> 另外 Navigation 组件配合 `mode(NavigationMode.Auto)` 可在折叠屏展开时自动切换为分栏模式，这是折叠屏适配的关键能力。

### 7.1 ArkUI 渲染三棵树

ArkUI 内部维护三层树结构来描述和渲染 UI：

```
Component Tree（组件树）
    开发者定义的组件层级
         │
         ▼
Element Tree（元素树）
    框架内部创建的中间表示，管理组件实例和状态
         │
         ▼
Render Tree（渲染树）
    最终提交给渲染引擎的绘制节点
```

这三层分离允许框架在 rebuild 时只重新生成 Render Tree，而不是重新创建整个组件树。这也是 `if/else` 切换时成本高的原因——需要修改三层树结构。

### 7.2 线程模型

鸿蒙采用 Actor 并发模型，核心原则是线程间内存隔离：

- **主线程（UI 线程）**：执行 build()、事件处理、布局计算。必须保持轻量。
- **TaskPool**：系统管理线程池，用于异步任务，8 个并发上限。
- **Worker**：手动创建的独立线程，有独立 JS 运行环境。最多创建 8 个。

线程间通信必须通过消息传递（Sendable 对象序列化）。这从架构上避免了传统多线程的锁和竞态条件问题，但传大数据时的序列化开销需要关注。

### 7.3 应用沙箱和文件路径

鸿蒙应用运行在沙箱中，文件系统分为：

```
应用沙箱
├── /data/storage/el1/base/    # 设备级加密区（锁屏后不可访问）
├── /data/storage/el2/base/    # 用户级加密区（锁屏后可访问）
│   ├── hap/                   # HAP 包资源
│   └── files/                 # 应用私有文件
├── /data/storage/el3/base/    # 更高安全级别
└── /data/storage/el5/base/    # 最高安全级别
```

应用通过 `context.filesDir`、`context.cacheDir`、`context.tempDir` 获取各级目录路径，不应对路径硬编码。
